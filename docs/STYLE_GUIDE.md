# 장표 스타일 가이드 (HTML Deck)

ArtPrep 문서 저장소에서 만드는 HTML 슬라이드 deck의 작성 규칙. 외부 발표·지원사업 제출 2가지 용도에 공통 적용되는 토대와 용도별 변종을 같이 정리.

> 본 가이드의 베이스는 사용자 전역 `CLAUDE.md` "Plan 산출물" 항목. 여기서는 art-prep 특화 항목(브랜드 컬러·로고·파일 위치·용도별 변종·텍스트 조판)만 덧붙임.

---

## 1. 적용 범위

이 가이드를 따르는 장표:

| 카테고리                              | 위치                                                | 예시                                                |
| ------------------------------------- | --------------------------------------------------- | --------------------------------------------------- |
| **외부 발표용 deck**            | `docs/meetings/<organization>/`                   | `boaz-intro-brief.html`, `boaz-intro-full.html` |
| **지원사업 제출용 PT**          | `docs/application/<program>/slides.html`          | `modoo/slides.html`                               |
| **지원사업 제출 이미지 추출용** | `docs/application/<program>/images/source/*.html` | `modoo/images/source/01-problem.html` 등          |

빠르게 끝나는 1~2 슬라이드 짜리 메모는 그냥 마크다운으로 작성. HTML deck은 5장 이상이거나 시각화·인쇄·외부 공유가 필요할 때만.

## 2. 공통 원칙 (모든 deck 공통)

### 2.1. self-contained

- **외부 CSS·JS·폰트 CDN 의존성 최소화.** 네트워크 끊겨도 작동해야 함.
- 폰트는 시스템 fallback으로 처리. Pretendard CDN을 쓸 경우엔 fallback도 반드시 작성.
- 인라인 `<style>` 한 곳에 CSS 다 모아두기.
- 이미지는 같은 디렉터리 또는 상대경로. 외부 URL 가급적 X.

### 2.2. 폰트 fallback (필수)

```css
font-family: 'Pretendard Variable', Pretendard, -apple-system, BlinkMacSystemFont,
             system-ui, 'Apple SD Gothic Neo', 'Noto Sans KR', 'Malgun Gothic',
             sans-serif;
```

Pretendard가 없는 환경(Windows·정부 평가관 PC 등)에서도 한국어가 깨지지 않게 Apple SD Gothic Neo → Malgun Gothic까지 명시.

### 2.3. 슬라이드 카드 헤더

각 슬라이드 카드에 다음 3요소 권장:

- **eyebrow** (작은 라벨) — 섹션·테마 표기. 색상은 accent 컬러.
- **title** (큰 제목) — 한 줄 요약. 56~84px (1920×1080 기준).
- **slide number chip** — `N/M` 형식. 우측 상단 또는 좌측 하단.

CLAUDE.md 규칙: "각 카드 헤더에 `SLIDE N/M` chip" — 다른 표기(예: `01 / 06`)도 허용.

### 2.4. 기본 비율: 16:9 1920×1080

**모든 장표의 기본은 16:9 프레젠테이션 비율 (1920×1080).** 화면 발표·PDF·PNG 출력에 공통으로 잘 맞음.

### 2.5. 슬라이드 가장자리 여백 (필수)

1920×1080 슬라이드의 바깥 padding은 **상하 80~100px / 좌우 100~120px**. 콘텐츠가 가장자리에 닿지 않도록.

```css
.slide {
  width: 1920px; height: 1080px;
  padding: 80px 120px;       /* 상하 80, 좌우 120 */
  /* 또는 padding: 100px 120px / padding: 56px 100px 100px */
}
```

발표 환경(빔프로젝터·TV)에서 화면 잘림 (overscan) 대비, 그리고 PDF 인쇄 마진 안정성을 위해 절대 잘라먹지 말 것.

### 2.6. 인쇄 호환

PDF 출력·PNG 캡처가 필요한 경우 슬라이드별 page-break:

```css
@media print {
  .slide { page-break-after: always; }
}
/* 또는 모든 환경 */
.slide { page-break-after: always; }
```

지원사업 제출용은 PDF 변환을 전제로 위 두 줄을 항상 포함.

### 2.7. 슬라이드 수

CLAUDE.md 기준: **8장 내외**. 외부 발표는 6~10장, 지원사업 제출은 5~8장 권장.

## 3. 브랜드 컬러 & 로고 (art-prep)

> **디자인 토큰 소스**: 실서비스 [art-prep.vercel.app](https://art-prep.vercel.app/) shadcn/ui slate 디폴트 + 파란 강조 (`#195AE6`). 단일 팔레트로 통일.

### 3.1. 컬러 팔레트 (서비스 UI 기준)

```css
/* 베이스 */
--ap-background: #ffffff;       /* 메인 배경 (라이트) */
--ap-foreground: #0f172a;       /* 본문 텍스트 */
--ap-primary: #0f172a;          /* 주요 액션 (다크 슬레이트) */
--ap-primary-fg: #f8fafc;       /* primary 위 텍스트 */

/* 보조 */
--ap-secondary: #f1f5f9;        /* 보조 배경 (소프트 슬레이트) */
--ap-muted: #f1f5f9;            /* muted 영역 */
--ap-muted-fg: #64748b;         /* 보조 텍스트·캡션 */
--ap-border: #e2e8f0;           /* 구분선·테두리 */

/* 강조 */
--ap-accent: #195AE6;           /* art-prep 파란 강조 (CTA·링크·하이라이트) */
--ap-accent-soft: #eff6ff;      /* 강조 배경 톤 */

/* 시멘틱 */
--ap-good: #16a34a;
--ap-warn: #d97706;
--ap-bad: #dc2626;
```

서비스 UI와 같은 톤이라 스크린샷이 그대로 박혀도 자연스럽고, 발표 환경에서도 가독성 안정적. 외부 발표·지원사업 제출·이미지 추출 **모두 이 팔레트만 사용.** 

### 3.2. 로고

**정식 art-prep 워드마크는 텍스트 렌더링.** 실서비스 (`c:/claude-class/art-prep/src/components/Header.jsx`) 와 동일한 마크업을 그대로 사용. 이미지(PNG/SVG)는 사용하지 않음. 해상도 깨짐·인쇄 픽셀라이즈 모두 회피되고, 컬러·굵기 조정이 자유로움.

```html
<!-- 정식 워드마크: 텍스트로 렌더링 -->
<span class="ap-wordmark">
  <span class="ap-art">Art</span><span class="ap-prep">PREP</span>
</span>
```

```css
.ap-wordmark {
  font-style: italic;
  color: var(--ap-accent);     /* #195AE6 */
  letter-spacing: -0.01em;
  white-space: nowrap;
  line-height: 1;
}
.ap-art  { font-weight: 500; }   /* medium */
.ap-prep { font-weight: 800; }   /* extrabold */

/* 사용 위치별 크기 */
.ap-wordmark.size-header  { font-size: 28px; }   /* 본문 슬라이드 헤더 */
.ap-wordmark.size-cover   { font-size: 96px; }   /* 표지 */
.ap-wordmark.size-small   { font-size: 20px; }   /* 푸터·캡션 */
```

- **워드마크 외 별도 마크 없음.** favicon에 있는 추상 도형은 임시이며 정식 마크가 아님.
- 다크 배경 위에서는 컬러를 흰색(`#ffffff`)으로 오버라이드 가능. 그 외에는 무조건 `#195AE6`.
- 사이즈 기준: 본문 슬라이드 헤더 28px, 표지 80~96px, 푸터·캡션 18~20px.
- **`docs/assets/logos/art-prep-logo.png` 는 deprecated.** 신규 deck 에서는 사용 금지. 기존 deck 도 점진적으로 텍스트 워드마크로 교체.

## 4. 용도별 변종

> 모든 변종 기본 사이즈 **1920×1080**, 컬러 §3.1. 차이는 레이아웃·네비게이션·인쇄 우선순위.

### 4.1. 외부 발표용 deck

- **사이즈**: `1920 × 1080`
- **레이아웃**: `display: none` + `.active` 토글로 한 번에 1장씩 표시 (발표 도구 스타일). 세로 스크롤 형 §4.2 방식도 허용.
- **네비게이션**: 없어도 됨 (키보드 화살표 키만으로 동작하면 충분). 굳이 넣을 경우 하단 고정 chip (`◀ N/M ▶`) 형식
- **인쇄**: 보조 — 화면 발표 우선, PDF는 부수적

### 4.2. 지원사업 제출용 PT

- **사이즈**: `1920 × 1080`
- **레이아웃**: **세로 스크롤로 모든 슬라이드 연속 표시** + `page-break-after: always`
  - **무슨 뜻?** 발표 도구처럼 한 장씩 토글이 아니라, 1920×1080 카드를 위에서 아래로 길게 이어붙인 웹페이지 형태. 브라우저 "인쇄 → PDF로 저장" 누르면 각 카드가 `page-break-after` 덕분에 PDF 1페이지로 깔끔히 잘림. 보는 사람은 스크롤로, 제출은 PDF로 — 두 출력을 한 HTML로 다 만족.
- **네비게이션**: 없음 (PDF 출력이 목표)
- **인쇄**: 필수 — `page-break-after: always` 반드시 포함
- **슬라이드 번호**: 우측 상단 chip

### 4.3. 지원사업 이미지 추출용 (modoo/images/source/*.html)

- **사이즈**: `1920 × 1080` (PNG로 캡처해 첨부 이미지로 사용)
- **레이아웃**: 1 파일 = 1 슬라이드. 스크린샷 자르기 쉽게 `overflow: hidden`
- **네비게이션**: 없음
- **인쇄**: 불필요 (목표는 PNG 캡처)
- **파일명**: `NN-<topic>.html` (예: `01-problem.html`, `02-solution.html`)
- **결과 PNG**: 상위 폴더 `images/` 직속에 저장. `.gitignore` 처리

## 5. 파일·디렉터리 컨벤션

### 5.1. 파일명

- 모두 lowercase-kebab. 한국어 X.
- 외부 발표: `<주체>-<주제>.html` (예: `boaz-intro-brief.html`)

- 이미지 소스: `NN-<topic>.html` (NN은 2자리 슬라이드 번호)

### 5.2. 디렉터리 위치

```
docs/
├── assets/
│   └── logos/
```

## 6. 이미지 첨부 규칙

장표에 이미지(스크린샷·다이어그램·사진·로고 등)를 넣을 때:

### 6.1. 파일 형식·위치

| 자산                                          | 형식                              | 추적 여부                                                       | 위치                                                                                       |
| --------------------------------------------- | --------------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 결과 출력 PNG (이미지 추출용 HTML의 산출물)   | `.png`                          | **`.gitignore`**                                        | `docs/application/<program>/images/*.png`                                                |
| 다이어그램·아이콘 (장표 본문 안 박힌 이미지) | `.svg` 우선                     | git 추적                                                        | 같은 폴더의 `assets/` 또는 인접 디렉터리                                                 |
| 사진·스크린샷                                | `.jpg`/`.png`                 | **`.gitignore`** (`.gitignore`의 `*.png` 룰 그대로) | 같은 폴더                                                                                  |
| **art-prep 로고/워드마크**              | `.png` (현재) / `.svg` (미래) | git 추적 (`.gitignore` 예외)                                  | [`docs/assets/logos/art-prep-logo.png`](./assets/logos/art-prep-logo.png) — 단일 진실 소스 |

### 6.2. 참조 방식

- **상대 경로만 사용.** 외부 URL은 절대 금지 (오프라인·발표 PC에서 깨짐).
- 이미지 추출용 HTML 안에서는 자체적으로 그라데이션·SVG로 그리는 것이 가장 안전 (외부 파일 의존성 0).
- `<img>` 태그는 `alt` 속성 필수.

### 6.3. 해상도

- 1920×1080 슬라이드에 들어가는 이미지: **원본 해상도 ≥ 표시 영역 × 2** (Retina 대응).
- 전체 화면을 채우는 배경 이미지: 최소 1920×1080, 가능하면 3840×2160.
- 다이어그램은 SVG로 만들면 해상도 무관.

### 6.4. 여백·세이프존

- 모든 이미지는 슬라이드 가장자리에서 **최소 80px 여백** 확보 (1920×1080 기준).
  - §2.5 슬라이드 자체 padding을 보통 `80px 120px`로 설정. 이미지가 padding을 침범하지 않게.
- 이미지를 카드 안에 넣을 때는 카드 내부에서도 상하좌우 **최소 24px** inner padding.
- 텍스트와 이미지가 함께 있을 때 둘 사이 간격 **최소 32px**.

### 6.5. 가독성

- 이미지에 텍스트가 들어 있으면 화면(또는 PDF)에서 18px 이상으로 보이는지 확인.
- 다크 배경 위 이미지는 외곽에 1px subtle border (`rgba(15,23,42,0.08)`) 또는 soft shadow로 분리.

### 6.6. 이미지 잘림 방지 — max-height·fit 결정법

슬라이드는 `overflow: hidden`이므로 콘텐츠 합이 1080을 넘으면 **하단부터 조용히 잘린다.** 이미지 영역이 큰 슬라이드(스크린샷 그리드 등)는 `max-height`를 적당히 큰 값(480·560)으로 두면 안전해 보여도, 비율 큰 이미지에서 그대로 그려져 슬라이드 영역을 침범한다. 콘텐츠 합을 직접 계산해 가용 영역에 맞춰라.

#### 6.6.1. 이미지 `max-height` 계산법

```
가용 영역 = 1080
         − slide padding (상+하, 보통 200)
         − header-block (margin-top + eyebrow + title + subtitle + margin-bottom)
         − card/shot padding (상+하)
         − caption 영역 (margin + title + sub)
```

예시 — SLIDE 5 (제품 스크린샷 4장 그리드):

| 항목 | px | 비고 |
| --- | --- | --- |
| slide padding | 200 | 100 + 100 |
| header-block | 382 | margin 100 + eyebrow 34 + title 2줄 156 + subtitle 1줄 56 + margin 36 |
| shot card padding | 52 | 24 + 28 |
| caption 영역 | 60 | margin 20 + title 20 + margin 4 + sub 16 |
| **잔여** | **386** | → `img max-height: 380px` (여유 6px) |

콘텐츠 양이 늘어나면(subtitle 2줄·title 3줄 등) max-height를 같은 식으로 다시 산출. 슬라이드 카피를 수정할 때마다 콘텐츠 합 재점검.

#### 6.6.2. `object-fit: contain` 기본

이미지 비율 보존이 중요한 경우(스크린샷·다이어그램). 카드 폭보다 좁아도 가운데에 비율 유지하며 표시. 한 행에 여러 이미지를 **비교 노출**할 때 보통 contain.

#### 6.6.3. `object-fit: cover` + `object-position: top` 예외

이미지 원본 비율이 카드 표시 영역 비율과 크게 어긋날 때(예: 매우 긴 세로 스크린샷·페이지 스크롤 캡처) contain으로 표시하면 카드 안에 좁고 길게 떠 시각 균형이 깨진다. 이때 `object-fit: cover` + `object-position: top`으로 박스를 채우고 위쪽 핵심 정보만 노출.

판단 기준:

- 이미지 원본 비율이 카드 표시 영역 비율과 **50% 이상 차이** → cover 검토
- 핵심 정보가 이미지 상단에 있는 경우만 `object-position: top` 적용 (리포트·랜딩페이지·요약 화면 등)
- **한 슬라이드 안 다른 이미지와 fit 방식 혼용은 최대 1장만.** 강조 효과는 있지만 일관성이 깨지므로 사용 빈도 제한

참고 구현 (SLIDE 5):

```html
<!-- 다른 3장: contain (전역 .shot img 룰 사용) -->
<img src="images/mvp-home.png" alt="...">

<!-- 1장만 cover top — 비율 0.226의 긴 세로 리포트 캡처 -->
<img src="images/mvp-report.png" alt="..."
     style="object-fit: cover; object-position: top;">
```

## 7. 텍스트 조판

장표는 결국 "정보 위계가 한눈에 보이는가"로 평가됨. 다음 원칙을 따른다.

### 7.1. 위계: 4단계로 충분

한 슬라이드에 보통 다음 4단계까지면 충분:

| 위계                    | 용도                 | 사이즈 (1920×1080) | 굵기                                                 |
| ----------------------- | -------------------- | ------------------- | ---------------------------------------------------- |
| **Eyebrow**       | 섹션/카테고리 표기   | 18~22px             | 600 (semibold), 글자 간격 `letter-spacing: 0.15em` |
| **Title (H1)**    | 슬라이드 핵심 메시지 | 56~84px             | 700~800 (bold/extrabold),`letter-spacing: -0.02em` |
| **Subtitle / H2** | 보조 설명·소제목    | 28~36px             | 600 (semibold)                                       |
| **Body**          | 본문·리스트         | 20~26px             | 400~500 (regular/medium),`line-height: 1.5~1.7`    |
| **Caption**       | 출처·각주·메타     | 14~16px             | 400,`--ap-muted-fg` 톤                             |

5단계 이상 위계가 필요하면 슬라이드를 2개로 나누는 게 낫다.

### 7.2. 근접성 원칙 (게슈탈트)

**관련된 내용끼리는 가깝게, 관련 적은 내용끼리는 멀게 배치.**

예시:

- **제목 1과 본문 1의 간격은 본문 1과 제목 2의 간격보다 짧아야 한다.**
- 같은 카테고리 bullet 간 간격은 12~16px, 카테고리 사이 간격은 32~48px.
- 한 카드 안에서 위계는 가깝게(eyebrow→title 8~12px / title→subtitle 16~24px), 다른 그룹 사이는 멀게(40~64px).

```
[제목 1]
  ↓ 12px       ← 가깝게 (제목 1과 본문 1은 한 묶음)
[본문 1]
  ↓ 48px       ← 멀게 (본문 1과 제목 2는 다른 묶음)
[제목 2]
  ↓ 12px
[본문 2]
```

CSS로는 묶음별로 `<div class="group">`을 만들고 `gap` 또는 `margin-bottom`을 위계 vs 그룹 간 차등화.

### 7.3. 정렬

- **본문 텍스트는 좌측 정렬 기본.** 가운데 정렬은 표지·1줄 핵심 메시지에만.
- 숫자·통계·표는 우측 정렬 또는 소수점 기준 정렬로 비교 가독성 확보.
- 한 슬라이드 안 정렬 방식 혼용은 의도가 있을 때만.

### 7.4. 텍스트 길이

- 한 줄 길이: 한국어 **20~40자**가 가독성 sweet spot. 50자 넘으면 줄바꿈 검토.
- 한 슬라이드 총 글자수: 한국어 **120자 내외 권장**, 250자 넘으면 슬라이드 분리.
- bullet 1개 = 한 문장 (또는 명사구). 두 문장 이상이면 본문으로 빼거나 둘로 쪼개기.

### 7.5. 강조

- 한 슬라이드에 **강조는 1~2개만.** 굵기·컬러·크기 중 하나로 통일.
- 색 강조는 §3.1의 `--ap-accent` (#195AE6) 또는 `--ap-foreground` (다크).
- 핵심 숫자(132분·22% 같은)는 `--ap-foreground` 굵게 + 인접 단위는 `--ap-muted-fg`로 톤 다운.

### 7.6. 공백 (white space)

- "비어 있는 공간"이 텍스트만큼 중요. 슬라이드를 가득 채우지 말 것.
- 1920×1080 슬라이드에서 콘텐츠 영역(padding 제외)은 1680×920. 이 중 **40% 이상은 공백으로 두는 것이 기본**.

### 7.7. 한국어 타이포 디테일

- 본문에 `word-break: keep-all`로 어절 단위 줄바꿈.
- 따옴표는 한국어 직선("...") 또는 둥근(『』·「」) 일관 사용. 영문 곡선(") 혼용 X.
- 숫자는 가독성 위해 천 단위 콤마 또는 한자 단위(만·억) 일관 사용.

### 7.8. 부호 사용 규칙

- **본문 한국어에 `—` (em dash) · `–` (en dash) 사용 금지.** 한국어 타이포에서 어색하고, 폰트별 렌더 폭 편차도 큼.
  - 의미 연결: 콤마(`,`) · 마침표(`.`) · 콜론(`:`) · 줄바꿈 중 하나로.
  - 부제·동격 표현: 콜론(`:`) 또는 `·` (중점) 사용. 예: `MVP 라이브: 사전신청 125명` / `솔루션 · 3축 모트`.
- 일반 하이픈(`-`)도 단어 연결(예: `B2B`, `AI-first`) 외에는 자제. 문장 부호로는 쓰지 않음.
- 화살표는 `→` (HTML 엔터티 `&rarr;`) 통일. ASCII `->` 또는 `=>` X.
- 중점(`·`)은 한국어와 잘 어울리며 짧은 나열·동격에 OK. 다만 한 문장에 3개 넘으면 분리.
- **본문이 아닌 부분에서는 마침표 쓰지 말 것.** title · eyebrow · subtitle · bullet · 카드 헤더 · caption · 표 셀 · chip · CTA 버튼 같은 단문/조각 텍스트는 마침표 없이 끝낸다. 마침표는 두 문장 이상의 본문 단락에서만.
  - 예: `MVP 라이브: 사전신청 125명` O / `MVP 라이브: 사전신청 125명.` X
  - 두 문장이 한 bullet에 들어가야 한다면 그것은 본문이므로 마침표 사용 OK. 단 §8.4 (bullet = 한 문장) 기준으로 두 문장이면 분리 검토.

### 7.9. 고립행 (widow) 금지

- **마지막 줄에 단어 1개만 남는 widow 금지.** 슬라이드 카피 가독성을 깎고 "AI가 자동 줄바꿈한 티"가 가장 많이 나는 부분.
- 점검 방법: 모든 다중행 텍스트(`title`, `subtitle`, 카드 `p`, `body`)를 슬라이드 폭에 렌더했을 때 마지막 줄 글자 수 확인.
  - 마지막 줄에 어절 1개 또는 5자 미만의 짧은 단어만 남으면 widow 로 간주.
- 처리 방법 (우선순위 순):
  1. 카피를 다시 쓴다 (제일 좋다). 한 단어 줄여 줄바꿈 위치를 옮긴다.
  2. 의도된 줄바꿈을 직접 넣는다 (`<br>`).
  3. 마지막 두 단어 사이에 줄바꿈 금지 — `&nbsp;` 또는 `<span style="white-space: nowrap;">` 으로 묶기.
- 슬라이드 표지·핵심 메시지 1줄은 **수동 줄바꿈 (`<br>`) 으로 의도된 라인을 직접 잡는다.** 자동 줄바꿈에 맡기지 않음.

### 7.10. 카드·블록 정렬

같은 행에 나란히 놓이는 카드·블록은 **높이가 같아야 한다.** 한쪽만 짧으면 시각적 균형이 깨지고, 평가자 시선이 빈 공간에 머문다.

- **그리드 정렬**: `display: grid` + `align-items: stretch` (기본값이지만 명시) → 자식 셀이 자동으로 같은 높이.
- **flex 정렬**: 부모에 `align-items: stretch`. 자식 카드에 `height: 100%` 또는 `flex: 1`.
- 카드 내부에서 콘텐츠 양 차이는 `margin-top: auto` 로 푸터 영역(이미지·CTA)을 아래로 밀어 정렬.
- 카드 간 높이 차이가 의도된 경우(예: 강조 카드 1장만 크게)에만 예외. 그 외에는 무조건 동일.

```css
/* 같은 행 카드 동일 높이 */
.grid-2, .grid-3, .grid-4 {
  display: grid;
  align-items: stretch;          /* 명시 */
}
.grid-2 > *, .grid-3 > *, .grid-4 > * {
  display: flex;
  flex-direction: column;
  height: 100%;                  /* grid 자식이 부모 높이 채우게 */
}
```

다른 콘텐츠를 묶어 한 쪽에 배치하는 경우(예: 좌측은 표/리스트, 우측은 설명 카드 1장)는 콘텐츠 양이 비대칭이라 `align-items: stretch`만으로 부족하다 → §7.11의 "컨테이너 조합별 정렬 규칙"을 따른다.

### 7.11. 컨테이너 정렬 — 텍스트 상단 고정, 박스만 stretch

§7.10이 "같은 행 카드 높이를 맞춘다"라면, §7.11은 **"그 높이를 어떻게 채우는가"**. 결론은 한 줄: **박스는 맞추되 텍스트는 항상 상단에.**

#### 7.11.1. 텍스트 상단 정렬 (필수)

카드·컨테이너 내부에서 `justify-content: center` / `justify-content: space-between`으로 텍스트를 가운데로 모으거나 위아래로 펼치지 말 것. 항상 `justify-content: flex-start` (기본값) 유지. 콘텐츠가 짧으면 카드 **하단**에 자연스러운 빈 공간이 남는다 — 그 빈 공간이 정상이다.

금지 패턴:

- **짧은 카드 → 가운데 정렬**해서 위·아래 균등 여백 만들기 (`.card { justify-content: center }`)
- **리스트 3개 → `space-between`**으로 펼쳐서 행 높이만큼 채우기 (마치 padding을 키운 것처럼 보임)

이유: 정렬로 만든 빈 공간은 시선 흐름을 흐트린다. 정보 위계는 항상 위에서 아래로 흘러야 하므로 텍스트는 무조건 상단 기준.

#### 7.11.2. 컨테이너 조합별 정렬·여백 규칙

같은 행에 놓이는 두 컨테이너의 **조합**에 따라 행 정렬과 짧은 쪽 처리 방식이 달라진다.

| 조합 (좌 + 우) | 행 정렬 `align-items` | 짧은 쪽 박스 처리 | 내부 텍스트 정렬 |
| --- | --- | --- | --- |
| 카드 × N, 콘텐츠 양 비슷 | `stretch` (기본) | 자식 `height: 100%` (글로벌 룰 유지) | `flex-start` |
| 카드 + 카드, 콘텐츠 양 다름 | `start` | 짧은 카드에만 `align-self: stretch` | 양쪽 모두 `flex-start` |
| 리스트(`.flex-col`) + 카드 | `start` | 리스트 `height: auto`, 카드 `align-self: stretch` | 양쪽 모두 `flex-start` |
| 이미지 그리드 + 텍스트 카드 | `start` | 텍스트 카드 `align-self: stretch` | 텍스트 카드 `flex-start` |
| 타임라인 + 카드 | `start` | 타임라인 `height: auto`, 카드 `align-self: stretch` | 양쪽 모두 `flex-start` |

핵심 원칙 3가지:

- **`stretch` 행 정렬은 콘텐츠 양이 거의 같을 때만.** 양 차이가 크면 `start`로 바꾸고, 짧은 쪽만 선택적으로 stretch.
- **`align-self: stretch`는 박스 높이만 늘린다.** 내부 텍스트 위치는 변동 없이 상단.
- **`.flex-col` 컨테이너 안에 `justify-content: space-between` 금지.** market-tier·timeline·bullet 그룹이 자동으로 펼쳐지면 행 간격이 들쭉날쭉해지고 §7.2 근접성이 깨진다.

#### 7.11.3. 빈 공간 채우는 순서 (패딩 vs 정렬)

콘텐츠 양이 부족해 박스가 작아 보일 때 해결 우선순위:

1. **카피를 더 쓴다 (제일 좋다).** 슬라이드에 정보 부족이라는 신호일 수 있다.
2. **`align-self: stretch`로 박스 높이만 키운다.** 내부 텍스트는 상단 그대로.
3. **그 슬라이드에서만 카드 자체 padding을 키운다.** 단, 같은 슬라이드 내 다른 카드와 padding이 달라지면 시각적 일관성이 깨진다. `.card`/`.card-accent`의 **글로벌** padding(`32px 36px`)은 절대 건드리지 말 것.
4. **자식 간 `gap`을 키운다.** §7.2 근접성 원칙(가깝게/멀게)을 깨지 않는 선에서.

`justify-content: space-between`으로 자식들을 위아래로 펼쳐 공간을 채우는 방식은 **어느 케이스에서도 권장하지 않는다.**

#### 7.11.4. 참고 구현 (SLIDE 7 패턴)

리스트 + 카드 조합, 콘텐츠 양 비대칭, 우측 카드 박스만 좌측 하단까지 늘리고 텍스트는 상단 유지:

```html
<div class="grid-2 gap-24" style="align-items: start;">
  <div class="flex-col" style="height: auto;">
    <!-- market-tier 3개: 자체 padding으로 행 간격 잡힘, space-between 없음 -->
    <div class="market-tier">...</div>
    <div class="market-tier">...</div>
    <div class="market-tier">...</div>
  </div>

  <div class="card" style="align-self: stretch;">
    <!-- .card 기본 flex column + justify-content: flex-start → 텍스트 상단 정렬 -->
    <h4>...</h4>
    <p>...</p>
    <ul class="bullets">...</ul>
  </div>
</div>
```

요지:

- 그리드 부모에 `align-items: start` (행 정렬은 위 기준)
- 리스트는 `height: auto`로 글로벌 `height: 100%` 무력화
- 카드만 `align-self: stretch`로 박스 높이를 행 max(=리스트 길이)에 맞춤
- 어느 쪽도 내부 `justify-content`를 건드리지 않음 → 텍스트는 항상 상단

#### 7.11.5. 그리드의 슬라이드 영역 차지 — `.stretch` 클래스 사용 기준

§7.10·§7.11.2가 다루는 `align-items: stretch`(행 정렬, **카드 간 높이 통일**)와, 유틸 클래스 `.stretch`(`flex: 1`, **그리드 자체가 슬라이드 남은 영역을 차지**)는 서로 다른 메커니즘이다. 헷갈리면 카드 박스가 콘텐츠 대비 비대해진다.

| 상태 | 동작 | 적용 시점 |
| --- | --- | --- |
| `.stretch` 적용 | 그리드가 슬라이드 남은 공간 전부 차지 → 행 높이 커짐 → `align-items: stretch`로 카드 박스 전체 확대 | 콘텐츠 양이 슬라이드 영역을 자연스럽게 채울 때만 |
| `.stretch` 미적용 | 그리드가 콘텐츠 크기만큼만 → 카드 박스 = 콘텐츠 + 카드 padding (`32×36`) | 콘텐츠 양이 슬라이드 영역 대비 작을 때 (대다수 케이스) |

판단 기준:

- 카드 안 텍스트가 카드 상단부터 하단 padding 근처까지 자연스럽게 채워짐 → `.stretch` 유지 OK
- 카드 안 텍스트가 상단 1/3 정도만 차고 하단에 카드 콘텐츠보다 큰 빈 공간이 남음 → `.stretch` 제거. 빈 공간은 슬라이드 자체 하단으로 흘려보내는 게 §7.6 ≥40% 공백 룰에 부합

§7.11.1과의 관계 정리: **카드 안 빈 공간이 콘텐츠보다 작으면 정상 (텍스트 상단 + 자연 하단 여백). 콘텐츠보다 크면 카드 박스 자체를 줄여야 한다** — 후자가 `.stretch` 제거 케이스.

```html
<!-- 잘못된 패턴: 그리드가 슬라이드 영역 차지, 짧은 콘텐츠가 거대한 카드 안 위쪽에 떠 있음 -->
<div class="grid-3 stretch">
  <div class="card-accent"><h4>...</h4><p>2줄짜리 짧은 설명</p></div>
  <div class="card-accent"><h4>...</h4><p>2줄짜리 짧은 설명</p></div>
  <div class="card-accent"><h4>...</h4><p>2줄짜리 짧은 설명</p></div>
</div>

<!-- 권장: .stretch 제거 → 카드는 콘텐츠 크기, 슬라이드 자체 하단에 자연 공백 -->
<div class="grid-3">
  <div class="card-accent"><h4>...</h4><p>2줄짜리 짧은 설명</p></div>
  ...
</div>
```

행 내 카드 동일 높이는 `.grid-2/3/4`의 글로벌 `align-items: stretch`가 그대로 보장하므로, `.stretch` 유틸을 빼도 §7.10 룰은 깨지지 않는다.

## 8. 텍스트 내용 (메시지·카피)

§7이 "어떻게 보이는가"라면, §8은 "무엇을 쓰는가". 같은 분량이어도 메시지를 잘 잡으면 다음 슬라이드까지 청자가 들고 간다.

### 8.1. One slide, one idea

- **한 슬라이드는 한 핵심 메시지.** 두 메시지 섞이면 둘로 쪼개라.
- 슬라이드 제목(title)만 따로 읽었을 때 데크 전체 흐름이 보여야 한다. 제목들이 "현황 / 솔루션 / 결과"처럼 스토리 라인을 형성하도록 작성.
- 슬라이드 안 모든 bullet·이미지·차트는 그 한 메시지를 뒷받침해야 함. 곁가지는 삭제.

### 8.2. 동사·구체성

- **능동 동사** 우선. "분석됩니다" → "분석한다", "도입 검토 중" → "도입한다 / 도입 예정 / 도입 안 함" 중 하나로.
- 모호한 형용사(혁신적·획기적·세계 최초) 회피. 구체 사실(132분 → 10초, 전환율 22%)로 대체.
- 형용사 1개 + 숫자 1개를 짝지어라 ("효율적" 단독 X → "효율적 (탐색 시간 85% 단축)" O).

### 8.3. 톤: 약속 / 진행 / 결과 구분

같은 정보라도 시제·완료도를 명확히. 검수자가 가장 흔히 묻는 질문이 "지금 어디까지 됐어요?"라서 모호하면 신뢰가 깎인다.

| 상태 | 표현 | 예시 |
|------|------|------|
| 결과 (완료) | 단정·과거형·정량 | "사전신청 125명 확보 (전환율 22%)" |
| 진행 | 현재 진행형·진척도 | "특허 출원 절차 진행 중 (5월 착수)" |
| 약속 (계획) | 시점·조건 명시 | "26.07 특허 출원 / W7 합격작 100장 도달" |

"~할 수도 있다 / ~를 검토 중 / ~가 될 것 같다" 같은 회피 표현은 제거. 안 정해진 건 안 정해졌다고 단호하게 쓰거나 슬라이드에서 빼라.

### 8.4. 카피 길이

- **bullet 1개 = 한 문장 또는 명사구.** 한국어 30자 이내 권장. 50자 넘으면 두 줄.
- **표지·핵심 메시지 1줄**: 한국어 15자 이내가 인상에 남는다 ("연 5만명, 132분이 10초로").
- 슬라이드 안 총 단어는 §7.4 (한국어 120자 내외 권장) 와 같이 운영.

### 8.5. 숫자·고유명사 다루기

- **숫자가 들어가면 출처 (또는 시점)도 표기.** "전환율 22% (2026-03 랜딩페이지 기준)".
- 큰 숫자는 한국어 단위(만·억) 사용. "3,942,000,000원" → "3,942억원" / "39억 4,200만원".
- 단위 통일: 한 슬라이드에 "억" 과 "백만원"을 섞지 말 것.
- 약어·전문어 (`MAU`, `LTV`, `CAC`, `B2B`)는 **첫 등장 시 1회 풀이**, 이후 약어만.

### 8.6. 한국어·영어 혼용

- **기본은 한국어.** 영어는 (a) 고유명사 (Pretendard, Supabase), (b) 보편 약어 (AI, MVP, PRD, LTV), (c) 정착된 외래어가 안 되는 경우만.
- 일관성: "AI" 와 "인공지능"을 같은 데크에서 섞지 말 것. 약어로 통일.
- 코드·식별자는 monospace + 영어 그대로.

### 8.7. 검증 가능성

- 발표·심사에서 가장 흔히 받는 질문: **"그 숫자 어떻게 나왔어요?"**
- 모든 정량 주장에 근거 한 줄 메모 (각주 또는 caption). "월 부담 90% 절감 — 학원 월 200만원 vs 구독 29,900원 기준" 식.
- 검증 못할 표현은 슬라이드에서 빼라. 입증 비용이 너무 크면 보조 자료(별첨 노트)로 미루기.

### 8.8. CTA·다음 단계

- 마지막 슬라이드는 **청자가 무엇을 하면 좋겠는지**를 명확히. "검토 부탁드립니다" 같은 막연한 인사 X.
- 예: "5/22(수)까지 1차 피드백 1건 / 6월 초 파일럿 학원 1곳 소개" 식 구체 요청.

### 8.9. AI 티 안 나는 자연스러운 문장

AI가 쓴 글은 일정한 패턴이 있다. 그 패턴만 피해도 사람이 쓴 글에 가까워진다.

#### 빼야 할 것

- **공허한 접속사·부사 남발**: "또한", "더불어", "특히", "결국", "궁극적으로", "나아가", "한편", "사실상". 빼도 의미 안 변하면 빼라.
- **영어 직역체**: "~을 통해서", "~에 있어서", "~함으로써", "~를 위해서", "~에 다름 아니다", "~라는 점에서".
  - "AI를 통해 효율을 극대화" → "AI로 탐색 시간 85% 단축".
- **추상명사 연쇄**: "혁신적인 솔루션을 통한 효율성의 극대화" → "탐색 시간 132분 → 10초".
- **수식어 동의어 반복**: "탁월한·뛰어난·우수한·획기적인·차별화된"을 같은 데크에서 돌려쓰기 X.
- **마무리 상투구**: "정리하자면", "결론적으로", "이상으로", "도움이 되길 바랍니다", "감사합니다" 외의 인사말.
- **emoji 과잉**: 본문에 ✨🚀💡🎯 같은 장식 이모지 X. 카테고리 아이콘(§4·§8)으로 1~2개까지만.
- **균질한 문장 길이**: 모든 문장이 비슷한 길이로 매끈하면 AI 티 남. 짧은 문장 + 긴 문장을 섞어라.
- **과도한 병렬 3단**: "A뿐 아니라 B도, 더 나아가 C까지" 식. 2단이면 충분.
- **모범생 호명**: "여러분", "독자 여러분", "우리 모두".

#### 더하면 좋은 것

- **숫자·고유명사**가 들어가면 곧바로 사람 글로 들린다. 추상 단어보다 "125명", "5/15", "에브리타임", "프라이머 클럽"이 먼저.
- **한국어 어순·구어 리듬**: 토씨 두 개 연속 ("~의 ~를"), 같은 조사 반복은 풀어라. "사용자의 행동의 패턴" → "사용자가 어떻게 행동하는지".
- **약간의 사적 디테일**: "6년 컨설팅", "본업 동료/지인 데이터팀"처럼 누가·언제·어디서가 들어가면 진짜처럼 들린다.
- **단정과 한계 인정**: 단정할 곳은 단정하되, 모르는 건 모른다고 ("아직 미정", "W7에 재검토"). AI 글은 보통 모든 걸 아는 척한다.
- **사람의 짧은 비문**: bullet에서 동사·조사 생략 OK. "**125명 사전신청 — 전환율 22%**" 가 "사전신청자는 125명이며 전환율은 22%입니다"보다 낫다.

#### 자가 점검 한 줄

> "이 문장에서 숫자·고유명사·구체 동사 다 빼면 뭐가 남는가?"
> 추상 단어만 남으면 AI 글이다. 다시 쓰라.

## 9. 작성 체크리스트

장표를 닫기 전 확인:

### 9.1. 구조·사이즈

- [ ] 사이즈가 **1920×1080** (기본). 다른 사이즈면 명확한 이유 있음.
- [ ] **슬라이드 가장자리 padding이 상하 80~100px / 좌우 100~120px 이상** (§2.5)
- [ ] 슬라이드 8장 내외 (외부 발표는 6~10장 OK)
- [ ] 인쇄 호환 필요한 경우 `page-break-after: always` 명시
- [ ] 슬라이드 번호 chip이 모든 슬라이드에 있음

### 9.2. 컬러·로고

- [ ] §3.1 단일 팔레트만 사용. 보라/청록·네이비/코랄 등 별도 변종 X
- [ ] **art-prep 로고는 텍스트 워드마크 (`<span class="ap-wordmark">`)** 로 렌더 (§3.2). PNG 이미지 사용 금지
- [ ] 강조는 슬라이드당 1~2개만 (§7.5)

### 9.3. 텍스트 조판

- [ ] 위계가 4단계 이하 (eyebrow → title → subtitle → body, §7.1)
- [ ] **근접성 원칙**: 같은 그룹 간격(12~16px) < 다른 그룹 사이 간격(40~64px) (§7.2)
- [ ] 본문 좌측 정렬 (가운데 정렬은 표지·핵심 메시지만, §7.3)
- [ ] 한 슬라이드 총 글자수 120자 내외 (250자 넘으면 분리, §7.4)
- [ ] 콘텐츠 영역의 40% 이상 공백 (§7.6)
- [ ] 한국어 `word-break: keep-all` 적용 (§7.7)
- [ ] **본문에 `—` em dash 사용 0건** (§7.8). 콤마·콜론·중점·줄바꿈으로 대체
- [ ] **본문이 아닌 부분(title·eyebrow·subtitle·bullet·캡션·chip 등)에 마침표 없음** (§7.8)
- [ ] **고립행(widow) 없음** (§7.9). 마지막 줄에 단어 1개만 남지 않음. 표지·핵심 메시지는 수동 `<br>` 줄바꿈
- [ ] **같은 행 카드·블록 높이 동일** (§7.10). 한쪽이 짧으면 `align-items: stretch` + 자식 `height: 100%` 적용
- [ ] **콘텐츠 양이 비대칭인 조합** (§7.11.2): 행 정렬은 `align-items: start`, 짧은 쪽에만 `align-self: stretch`. 카드 × N으로 양이 비슷할 때만 `stretch` 행 정렬 사용
- [ ] **컨테이너 내부 텍스트 상단 정렬** (§7.11.1). 카드·flex-col 안에서 `justify-content: center`·`space-between`으로 텍스트를 분산시키지 않음. 콘텐츠 짧으면 카드 하단에 빈 공간 OK
- [ ] **`.stretch` 유틸 클래스는 콘텐츠가 슬라이드 영역을 자연스럽게 채울 때만** (§7.11.5). 카드 안 빈 공간이 콘텐츠보다 크면 `.stretch` 제거하고 빈 공간을 슬라이드 자체 하단으로 흘려보낼 것

### 9.4. 텍스트 내용 (메시지·카피)

- [ ] **한 슬라이드 한 메시지** — 제목들만 따라가도 데크 스토리가 보임 (§8.1)
- [ ] 모호한 형용사 → 구체 숫자·사실로 대체 (§8.2)
- [ ] 약속/진행/결과 톤 일관 (§8.3) — "~할 수도 있다" 같은 회피 표현 X
- [ ] bullet 1개 = 한 문장(30자 이내), 표지 1줄 메시지는 15자 이내 (§8.4)
- [ ] 모든 숫자에 출처·시점 표기 (§8.5)
- [ ] 약어 (`MAU`, `LTV` 등) 첫 등장 시 풀이 (§8.5)
- [ ] AI/인공지능 같은 표현 한 데크 내 통일 (§8.6)
- [ ] 마지막 슬라이드에 구체 CTA — 청자에게 무엇을 요청하는지 명확 (§8.8)
- [ ] **AI 티 안 남** (§8.9): "또한·더불어·궁극적으로" 등 공허 부사, "~을 통해·~에 있어서" 직역체, 추상명사 연쇄, 균질한 문장 길이 없음. 숫자·고유명사·구체 동사 다 빼도 의미가 남는가? — 남으면 다시 쓰기

### 9.5. 이미지·자산

- [ ] 모든 이미지가 슬라이드 가장자리에서 80px 이상 여백 확보 (§6.4)
- [ ] 이미지가 카드 안에 들어갈 때 카드 내부 24px 이상 inner padding (§6.4)
- [ ] 이미지·다이어그램의 텍스트가 18px 이상으로 보임
- [ ] `<img>` 태그에 `alt` 속성 있음
- [ ] **이미지 `max-height`가 슬라이드 가용 영역 안** (§6.6.1). 콘텐츠 합 = padding + header + card padding + caption + 이미지를 계산해 1080 이하 확인. 480·560 같은 적당히 큰 값을 그냥 두지 말 것
- [ ] **이미지 비율이 카드 표시 영역과 50% 이상 어긋나면 `object-fit: cover` + `object-position: top` 검토** (§6.6.3). 단 한 슬라이드 안 다른 이미지와 fit 혼용은 최대 1장

### 9.6. 기술 호환

- [ ] 폰트 fallback이 `Pretendard → Noto Sans KR → Apple SD Gothic Neo → Malgun Gothic` 포함
- [ ] 외부 의존성 없음 (또는 fallback 있음). 오프라인 환경에서 열어봤을 때 깨지지 않음
- [ ] 파일명·위치가 §5 규칙에 맞음
- [ ] 한국어가 깨지지 않음 (`<meta charset="UTF-8">` 확인)

원본은 그대로 두고, 색·텍스트만 교체. **단 기존 파일이 §3.1 단일 팔레트 이전 컬러(보라/청록·네이비/코랄)를 사용 중이면, 새 장표에서는 §3.1로 교체.** 구조를 크게 바꿔야 한다면 사용자에게 먼저 확인.

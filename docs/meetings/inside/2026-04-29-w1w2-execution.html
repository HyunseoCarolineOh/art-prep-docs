축 1 — 필터 옵션 SoT 이전 (코드 → DB)

filter_options 마스터 테이블 신설. category ∈ {university, art_type, academy_type}를 CHECK로 잠금, UNIQUE (category, value) + 시딩 ON CONFLICT DO NOTHING으로 멱등성, partial index (category, display_order) WHERE is_active = true로 핫패스 최적화. 정렬 키는 가나다순 ROW_NUMBER × 10에 '미지정'만 별도 그룹으로 빼는 CASE WHEN 조합 — "미지정"이 항상 맨 뒤, 사이사이에 새 옵션 끼워 넣을 자리(11, 15...) 확보.

FilterBar.jsx·UploadModal.jsx가 동일 쿼리로 fetch (현재는 중복, SWR/Context 도입 시점에 통합 예정). 로딩 중에는 select disabled + placeholder "불러오는 중..." → 사용자가 빈 select에 멘붕하지 않게.

축 2 — 저장한 작품 페이지 (로컬 상태 → 서버 상태)

핵심 쿼리는 외래키 임베딩으로 한 번에 작품 본체까지:

supabase
  .from("saved_items")
  .select("artwork_id, created_at, artworks(*)")
  .eq("user_id", user.id)
  .order("created_at", { ascending: false })

토글 핸들러는 optimistic UI → mutation → 실패 시 rollback. 단, 저장 해제 실패 시에는 단순 rollback이 아니라 작품 본체를 다시 fetch해서 목록에 넣어줘야 한다 — 이미 배열에서 빠진 객체를 직접 들고 있지 않기 때문. pendingSet(Set<id>)로 인플라이트 토글이 있는 동안 같은 작품의 신규 토글을 차단해 INSERT 직후 DELETE가 race로 도착하는 사고를 막음.

축 3 — 업로드 폼 (가짜 alert → 실제 업로드 + 보상 트랜잭션)

4단계 파이프라인: 클라 검증(MIME jpeg/png/webp · 5MB · 제목 1~50자 · 필수 select 3개) → Storage 업로드(artwork-images/{user.id}/{Date.now()}.{ext}) → public URL 추출 → artworks INSERT. INSERT 실패 시 방금 올린 Storage 파일을 즉시 remove([path]) — 외톨이 파일 방지. Storage + DB가 별개 시스템이라 양방향 트랜잭션 불가능, 클라이언트가 보상 트랜잭션 코디네이터.

경로 설계: user.id로 시작 → RLS 정책에서 auth.uid()::text = (storage.foldername(name))[1]로 본인 폴더만 쓰기 허용 가능. 원본 파일명 미사용 → 한글/공백/이상 문자 URL 인코딩 사고 회피.

UX: submitting 상태 동안 모든 입력 + 닫기 버튼 + Dialog 외부 클릭 모두 disabled (onOpenChange에서 if (submitting) return) → 업로드 도중 모달이 닫혀 멈춘 사고 방지.

축 4 — 본인 업로드 관리 (MyUploadsPage)

본인이 올린 작품만 모아 보고 그 자리에서 삭제. RLS가 막아주지만 클라이언트에서도 .eq("uploaded_by", user.id)로 한 번 더 명시 — defense in depth. 삭제 = 업로드의 역방향 보상 트랜잭션: confirm → DB row 삭제 → 성공 시 Storage 파일 삭제 → Storage 실패는 console.warn으로만 (외톨이 회수는 추후 cron). deletingId 단일 락으로 동시 삭제 배타.

UI는 마소너리가 아닌 고정 종횡비(aspect-[3/4]) 그리드 — "재고를 보는" 인지 모델에 더 맞음.

정리 작업 — ArtworkDetailModal 폐기 + 폼 시각화

이전엔 카드 클릭 시 모달이 뜨고, 모달 안에 또 페이지로 가는 경로가 있어 두 개의 상세 진입점이 공존했음. 모달을 통째로 지우고 /artwork/:id 페이지 단일 라우트로 일원화. HomePage.jsx의 selectedArtwork state · setSelectedArtwork 호출 · <ArtworkDetailModal> 렌더 모두 제거.

UploadModal 8필드 중 6개가 필수 — 제출 전까지 무엇이 필수인지 모르는 사용자 멘붕 방지. FormField에 required/optional prop 추가, 헤더에 별표 범례. 별도 커밋으로 분리한 이유는 시각 가이드만 빼고 싶을 때 1 commit revert로 가능하도록.

07 · TECH LESSONS

기술 학습 포인트 — 오늘의 4개 패턴

① 단일 진실 공급원(SoT)의 이동

필터 옵션의 SoT를 src/data/artworks.js 배열에서 filter_options 테이블로 이동. 비용: 마이그레이션 SQL · RLS 설정 · 클라 fetch + 로딩/에러 분기 · 시딩 멱등성. 이득: 운영자가 코드 배포 없이 옵션 변경 / 표기 일관성(서울대 vs 서울대학교 분기 차단) / 재실행 안전성 / is_active 토글로 노출 제어. SoT 이동의 손익은 운영 변경 빈도에 비례 — 옵션이 자주 바뀌는 마스터 데이터일수록 이득이 크다.

② 서버 상태 = 멱등성 + 보상 트랜잭션

오늘 두 곳에서 같은 패턴이 등장: 저장 토글(optimistic UI → mutation → 실패 시 rollback)과 업로드(Storage upload → DB insert → insert 실패 시 Storage 삭제). 클라이언트가 두 시스템(Storage / DB / UI)을 가로지르는 트랜잭션 코디네이터 역할을 하며, 어느 한쪽이 실패하면 다른 쪽을 원상 복구하는 코드를 직접 써야 한다. DB 단일 트랜잭션처럼 자동으로 해주지 않는다.

③ Race Condition 방어의 3가지 가드

pendingSet(Set<id>) — 동일 작품에 인플라이트 토글이 있으면 신규 토글 무시 / deletingId(단일 id) — 한 번에 한 작품만 삭제 / cancelled 플래그 + cleanup — useEffect 내부 fetch가 unmount 후 setState 호출하지 않도록. 세 가지 모두 React Strict Mode + 빠른 사용자 입력에서 흔히 발생하는 사고를 사전에 차단하는 가드.

④ 인증 강제 지점의 분포 — 의도적 삼중 가드

같은 정책을 세 군데서 강제: Header(업로드 버튼 클릭 시점에 즉시 로그인 페이지로) / Page mount의 useEffect(직접 URL 진입 케이스) / Toggle 핸들러(카드 클릭 진입점). 한 곳이 빠져도 다른 곳에서 막히도록 의도적 중복. "저장은 둘러볼 수 있는 동선, 업로드는 기여 행위라 인증이 전제"라는 정책 차이가 두 페이지의 가드 강도 차이로 이어짐.
